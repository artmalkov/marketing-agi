# Find Competitors Rulebook

## Document Metadata

- **Version:** 1.1.0
- **Creation Date:** 2026-03-14
- **Purpose:** Define rules for finding focused competitors for a student project, including region-aware search and latest news collection

---

## Purpose

This rulebook defines the process for finding focused competitors for a student project, collecting baseline market information, and saving the result to the `competitors/` folder in the student project root.

Use this workflow when the user writes requests such as:

- `найди конкурентов`
- `найди конкурентов для проекта`
- `найди конкурентов в СНГ`
- `найди конкурентов в Азии`
- `кто наши конкуренты в Европе`
- `find competitors`
- `find competitors for this project`
- `find competitors in CIS`
- `find competitors in Asia`
- `find competitors worldwide`

Target result:

- find **at least 10 focused competitors** in the requested geography
- if reliable competitors are fewer than 10, save the **maximum valid number found**
- keep **large conglomerates** out of the main competitor list and show them in a separate section
- save **latest news** for each competitor in a separate page inside that competitor's folder

## Step 1. Определение студенческого проекта (критично)

**Очень важно не ошибиться с выбором проекта.** Изменение чужого проекта может серьёзно навредить другому студенту.

### Способы определения целевого проекта

Используй **все доступные сигналы** и действуй по приоритету:

1. **Явно указанный путь или источник данных**
   - Пользователь указал путь к файлу: `student-projects/[student-name]/...` → проект определён
   - Пользователь сослался на файл из проекта: `genotype.csv`, `call-database.md`, `landing.html`, `competitors.md` → проект определён
   - Пользователь явно назвал проект или студента, и это имя однозначно соответствует папке в `student-projects/`

2. **Открытый в IDE файл**
   - Если в редакторе открыт файл из `student-projects/[student-name]/...`, считай этот проект целевым
   - Перед началом работы проверь, что файл действительно относится к нужному студенческому проекту

3. **Контекст запроса**
   - Пользователь пишет `найди конкурентов для моего проекта`, а в текущем контексте есть только один активный студенческий проект
   - Пользователь продолжает предыдущую работу по конкретному проекту
   - Пользователь прикрепил описание, landing, genotype или список конкурентов, уже относящиеся к одному проекту

### Если проект неочевиден

Если после проверки остаётся **больше одного кандидата** или нет ни одного надёжного сигнала:

- **не выполняй поиск**
- **не создавай файлы**
- спроси пользователя прямо: `Для какого проекта искать конкурентов? Укажи папку в student-projects или имя проекта.`

### Защита от ошибки

**Обязательные проверки перед любыми изменениями:**

- [ ] Определён **ровно один** проект
- [ ] Все пути для чтения и записи начинаются с `student-projects/[student-name]/`
- [ ] Целевая папка для результата: `student-projects/[student-name]/competitors/`
- [ ] Никогда не пиши в `student-projects/` без имени студента
- [ ] Если проект определён только предположительно, сначала запроси подтверждение

**Структура студенческих проектов:**
```text
student-projects/
  [student-name]/           ← имя папки = идентификатор проекта
    competitors/            ← целевая папка для списка конкурентов
    genotype/
      genotype.csv
      call-database.md
    webpages/
      landing.html
    customer-calls/
    employees/
```

## Step 2. Определение контекста продукта и области поиска

Перед поиском конкурентов собери **search brief**. Он должен зафиксировать:

- какой это проект
- что именно продаёт или делает продукт
- для кого продукт предназначен
- какую задачу или Job-To-Be-Done он закрывает
- в каком регионе нужно искать конкурентов
- есть ли уже известные конкуренты или похожие продукты
- какие типы решений нужно исключить

### 2.1. Источники контекста продукта

Используй **все релевантные источники**, а не только один.

#### Источник A: Явное описание от пользователя

Если пользователь сам описал продукт, это приоритетный источник. Извлеки:

- тип продукта
- целевую аудиторию
- ключевую задачу
- платформу или формат решения
- ограничения по рынку, языку или географии

#### Источник B: Подсказки через похожих конкурентов

Если пользователь прислал:

- названия конкурентов
- ссылки на сайты конкурентов
- формулировки вроде `похожие на X`, `аналоги Y`, `look for products like Z`

используй это как **seed list**:

- определи нишу по этим примерам
- найди прямых и близких игроков в той же категории
- добавь новых релевантных конкурентов сверх исходного списка
- не дублируй уже присланные компании без проверки их релевантности

#### Источник C: Материалы студенческого проекта

Если пользователь не дал полного описания, изучи проект:

- `student-projects/[student-name]/genotype/genotype.csv` — формулировки потребностей и сегменты
- `student-projects/[student-name]/genotype/call-database.md` — описание продукта, боли, возражения, цитаты клиентов
- `student-projects/[student-name]/webpages/landing.html` — позиционирование и ценностное предложение
- `student-projects/[student-name]/competitors/competitors.md` — если файл уже есть, используй как базу для обновления и расширения

#### Источник D: Комбинированный контекст

Если одновременно есть:

- описание пользователя
- файлы проекта
- seed-конкуренты

объединяй их. Приоритет смыслов такой:

1. Явные указания пользователя для текущей задачи
2. Файлы выбранного проекта
3. Seed-конкуренты как подсказка по нише

### 2.2. Когда нужно спросить пользователя ещё раз

Запроси уточнение, если:

- из описания непонятно, какой именно продукт искать
- продукт слишком общий и получается несколько разных ниш
- пользователь написал `найди конкурентов`, но неясно, для какого проекта это делать
- пользователь прислал названия конкурентов, которые относятся к разным рынкам

Вопрос должен быть коротким и предметным. Например:

- `Для какого проекта искать конкурентов?`
- `Какой именно продукт или категория у проекта?`
- `Ищем конкурентов глобально или в конкретном регионе?`

### 2.3. Определение географии поиска

Определи регион поиска из текста запроса. Агент должен понимать формулировки и на русском, и на английском.

Примеры:

- `найди конкурентов в СНГ` → регион = `CIS`
- `найди конкурентов в Азии` → регион = `Asia`
- `найди конкурентов в Европе` → регион = `Europe`
- `find competitors in LATAM` → регион = `LATAM`
- `find competitors worldwide` → регион = `Global`

Если пользователь указал конкретную географию, **ищи именно там**.

Если пользователь географию **не указал**, используй регион по умолчанию:

- `Global` / `Worldwide`

Если в проекте раньше исследовался один рынок, а пользователь сейчас явно указал другой, для текущего запуска приоритет у **текущего запроса пользователя**.

## Step 3. Поиск конкурентов

### 3.1. Что считается основным конкурентом

Основной конкурент для этого workflow — это продукт или компания, которые:

- решают **ту же основную задачу**
- ориентированы на **схожую аудиторию**
- находятся в указанной географии или реально работают в ней
- могут рассматриваться клиентом как реалистичная альтернатива

Приоритет всегда у **прямых конкурентов**. Смежные игроки допустимы только если прямых мало и это явно отмечено в примечаниях.

### 3.2. Что не включать в основной список

Не включай в основной список:

- обзорные сайты, рейтинги, маркетплейсы и каталоги
- агентства, интеграторов и консалтинг, если проект сам не является агентством
- нерелевантные инструменты, закрывающие только маленький подшаг задачи
- родительские компании больших концернов, если конкурентом выступает конкретный продукт

### 3.3. Правило для крупных концернов

Большие концерны вроде `Microsoft`, `Google`, `Amazon`, `Meta` и аналогичных **не должны** попадать в основной список как самостоятельные конкуренты только потому, что у них много продуктов.

Если у концерна есть релевантный продукт:

- в основной список добавляй **продукт**, а не родительский концерн
- в отдельный раздел `Крупные концерны с релевантными продуктами` добавляй сам концерн и его продукт

Пример:

- в основной список: `Google Meet`
- в отдельный раздел: `Google` → `Google Meet`

Родительский концерн **не засчитывается** как один из 10 основных конкурентов.

### 3.4. Источники и стратегия поиска

Используй доступные источники:

- поисковые запросы по нише, продукту и региону
- официальные сайты конкурентов
- отраслевые обзоры и рейтинги
- базы компаний и рынков, если они доступны
- новостные публикации и официальные пресс-релизы для последней новости

Строй поиск вокруг:

- описания продукта
- формулировок потребностей из проекта
- seed-конкурентов от пользователя
- региона поиска

Ищи запросами на русском и английском, если это помогает собрать более полный список.

Примеры поисковых паттернов:

- `[product category] competitors`
- `[product category] alternatives`
- `[product category] competitors in [region]`
- `[seed competitor] alternatives`
- `[JTBD / niche] software in [region]`

### 3.5. Целевое количество

Цель — собрать **не менее 10 валидных конкурентов**.

Если после разумного поиска найдено меньше 10:

- не придумывай компании
- не добавляй слабые и нерелевантные варианты ради количества
- сохрани **максимальное число реально подходящих конкурентов**, даже если это 8
- явно укажи в `competitors.md` и сообщении пользователю, что найдено меньше 10 и почему

### 3.6. Данные по каждому конкуренту

Для каждого конкурента собери:

- **Название** — обязательно
- **Сайт / URL** — обязательно
- **Регион** — основной рынок или география присутствия
- **Тип** — direct или adjacent
- **Выручка** — если доступно
- **Инвестиции** — если доступно
- **Краткое описание** — что делает продукт и чем релевантен
- **Последняя новость** — заголовок самой свежей релевантной новости
- **URL новости** — ссылка на новость
- **Примечания** — важные факты: сегмент, масштаб, pricing, пользователи, особенности
- **Бизнес-модель** - если доступно, как зарабатывает продукт(подписка, pay as you go и так далее)

### 3.7. Правила для последней новости

Для каждого конкурента найди **одну самую свежую релевантную новость**:

- статья в СМИ
- официальный пресс-релиз
- новость о запуске продукта, инвестициях, партнёрстве, росте, M&A или заметном обновлении

Если надёжной новости не найдено:

- в поле `Последняя новость` запиши `Не найдено`
- в поле `URL новости` оставь пусто
- в файле `latest-news.md` зафиксируй, что надёжная публичная новость не найдена

## Step 4. Сохранение результатов

### 4.1. Структура папок

```text
student-projects/[student-name]/
  competitors/
    competitors.md
    [competitor-slug]/
      latest-news.md
```

Если `competitors/` не существует, создай её **только внутри выбранного проекта**.

Если `competitors.md` уже существует:

- обновляй файл, а не создавай дубликат
- сохраняй полезные ручные заметки, если они не противоречат новым данным
- не дублируй конкурентов с тем же сайтом или тем же продуктом

### 4.2. Файл `competitors.md`

Используй структуру из `knowledge/competitor-analysis/data/templates/competitors-template.md`.

Минимальное содержимое:

- заголовок `Конкуренты`
- блок `Контекст продукта`
- блок `Параметры поиска`
- таблица `Основные конкуренты`
- отдельный раздел `Крупные концерны с релевантными продуктами`
- список созданных подпапок

### 4.3. Подпапка каждого конкурента

Для каждого конкурента создай подпапку:

- `student-projects/[student-name]/competitors/[competitor-slug]/`

Правила именования:

- латиница
- нижний регистр
- слова через `-`
- без пробелов и спецсимволов
- если конкурент в основном списке представлен конкретным продуктом концерна, имя папки должно отражать **продукт**, а не родительскую компанию

Примеры:

- `notion`
- `miro`
- `google-meet`

### 4.4. Файл `latest-news.md`

Внутри каждой папки конкурента создай файл:

- `student-projects/[student-name]/competitors/[competitor-slug]/latest-news.md`

Минимальная структура файла:

```md
# Latest News — [Competitor Name]

- Competitor: [Competitor Name]
- Search Scope: [Region]
- News Title: [Latest news title] / Не найдено
- Published At: [YYYY-MM-DD or empty]
- URL: [News URL]
- Why It Matters: [1-2 short lines]
```

Одна папка конкурента = один файл `latest-news.md` для самой свежей найденной новости на момент поиска.

---

## Step 5. Проверка перед сохранением

- [ ] Определён ровно один студенческий проект
- [ ] Все файлы создаются внутри `student-projects/[student-name]/competitors/`
- [ ] Регион поиска соответствует явному запросу пользователя, если он был указан
- [ ] В основном списке собраны 10+ конкурентов или сохранён максимум найденных валидных конкурентов
- [ ] Для каждого конкурента заполнены **Название** и **Сайт**
- [ ] Нет дубликатов по домену или одному и тому же продукту
- [ ] Крупные концерны вынесены в отдельный раздел, а не смешаны с основным списком
- [ ] Для каждого конкурента создана нормализованная подпапка
- [ ] Для каждого конкурента создан `latest-news.md`
- [ ] В `competitors.md` есть `URL новости`, если надёжная новость найдена

---

## Output

- **`competitors.md`** — сводный список конкурентов в выбранном студенческом проекте
- **`competitors/[competitor-slug]/latest-news.md`** — отдельная страница с последней новостью по каждому конкуренту
- **Сообщение пользователю:** какой проект выбран, какой регион использовался, сколько конкурентов найдено, были ли вынесены крупные концерны отдельно, где всё сохранено
