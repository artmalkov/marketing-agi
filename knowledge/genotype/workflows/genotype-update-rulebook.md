# Genotype Update Rulebook

## Document Metadata

- **Version:** 1.0.0
- **Creation Date:** 2026-03-05
- **Purpose:** Define rules for creating or updating a genotype CSV table from call-database.md — mapping client needs to interviews and polls with vote counts

---

## Purpose

This rulebook defines the process for creating or updating a **genotype table** — a CSV file that maps each client need (from call-database) to individual interviews and polls, with vote/confirmation counts per source.

The genotype is built from:
- **Input:** `call-database.md` in the student project's genotype folder
- **Template:** `knowledge/genotype/data/templates/genotype-template.csv`
- **Output:** `genotype.csv` in the same genotype folder (not .md — CSV format for Excel export)

---

## Related Concepts

- **Genotype Needs Taxonomy** (`knowledge/genotype/concepts/taxonomies/genotype-needs-taxonomy.md`) — defines need intensity levels T, M, D, N, U
- **Product Genotype** (`knowledge/genotype/concepts/concepts/product-genotype.md`) — demand vector (needs) vs supply vector (solutions)
- **Genotype Template** (`knowledge/genotype/data/templates/genotype-template.csv`) — template structure for the genotype table

## Step 1. Определение студенческого проекта (критично)

**Очень важно не ошибиться с выбором проекта.** Изменение чужого проекта может серьёзно навредить другому студенту.

### Способы определения целевого проекта

Используй **все доступные сигналы** и действуй по приоритету:

1. **Явно указанный путь или источник данных**
   - Пользователь указал путь к файлу: `student-projects/[student-name]/...` → проект определён
   - Пользователь сослался на файл (например, call-database.md, genotype) → путь к файлу однозначно определяет проект

2. **Открытый в IDE файл**
   - Если в редакторе открыт файл из `student-projects/[student-name]/...`, считаем этот проект целевым
   - Проверь путь открытого файла перед началом работы

3. **Контекст запроса**
   - Пользователь упомянул имя студента или проекта
   - Пользователь работал с данным проектом в предыдущих сообщениях

### Защита от ошибки

**Обязательные проверки перед любыми изменениями:**

- [ ] **Подтверждение:** Если проект не очевиден из одного источника — явно спроси пользователя: «Работаем с проектом `[student-name]`?»
- [ ] **Единственность:** Убедись, что определён ровно один проект. Если возможны два — не действуй, запроси уточнение.
- [ ] **Проверка пути:** Все пути к файлам должны начинаться с `student-projects/[student-name]/`. Никогда не пиши в `student-projects/` без имени студента.
- [ ] **Читай перед записью:** Перед изменением файла проверь, что он принадлежит выбранному проекту.

**Структура студенческих проектов:**
```
student-projects/
  [student-name]/           ← имя папки = идентификатор проекта
    genotype/
      call-database.md       ← источник потребностей (обязательный)
      genotype.csv           ← целевой файл (создавать/обновлять)
    customer-calls/
    employees/
```

## Step 2. Загрузка call-database и шаблона

### 2.1. Источник потребностей

**Обязательный файл:** `student-projects/[student-name]/genotype/call-database.md`

Если файл отсутствует — сообщи пользователю и не создавай genotype.csv.

### 2.2. Шаблон

**Шаблон:** `knowledge/genotype/data/templates/genotype-template.csv`

Используй структуру шаблона для формирования CSV. Структура:
- **Строки 1–4:** Заголовки (даты, имена/опросы, тип источника, Number/Budget/Need)
- **Строки 5+:** Секции потребностей и строки потребностей

---

## Step 3. Извлечение уникальных источников (колонки)

### 3.1. Парсинг Source Calls

Из каждого блока NEED в call-database извлеки все записи из **Source Calls**:

**Формат интервью:**
```
Call-YYYY-NNN - YYYY-MM-DD - [Название звонка] - [Имя клиента] - Client Direct Quote
```
или
```
Call-YYYY-NNN - YYYY-MM-DD - [Название звонка] - [Имя клиента] - Employee Statement...
```

**Формат опроса (Poll):**
```
Call-YYYY-NNN - YYYY-MM-DD - Опрос: «[вопрос]» — N человек
```
или с диапазоном: `3–4 человека`, `несколько человек`

### 3.2. Уникальные источники

Собери **уникальный список** источников по ключу:
- **Интервью:** `(Call-ID, Date, Client Name)` — один человек в одном звонке = одна колонка
- **Опросы:**
  - **Разные даты** — каждый опрос на свою дату = отдельная колонка (разные голосования)
  - **Одна дата, нет информации для различения** — все опросы с этой даты объедини в **одну колонку**. Не разбивай по отдельным вопросам.
  - **Одна дата, много разных вопросов с явной привязкой** — можно вынести в отдельные колонки, если есть информация, позволяющая их различать

Дедупликация: если Олег встречается в NEED-001 и NEED-002 — это одна колонка «Олег» (интервью).

### 3.3. Сортировка колонок

**Порядок:** от самого позднего к самому раннему (по дате).

- **Левая колонка** = последнее (самое недавнее) интервью/опрос
- **Правая колонка** = самое раннее интервью/опрос

При одинаковой дате — сохраняй порядок появления в call-database (интервью перед опросами, если применимо).

## Step 4. Заполнение заголовков (строки 1–4)

### 4.1. Строка 1 — Даты

| Колонка | Содержимое |
|---------|------------|
| 1–5 | Пусто (;;;;) |
| 6 | Пусто |
| 7+ | Дата каждого источника в формате YYYY-MM-DD (или кратко) |

### 4.2. Строка 2 — Имя клиента / Опрос

| Колонка | Содержимое |
|---------|------------|
| 1–5 | Пусто |
| 6 | Пусто |
| 7+ | Для интервью: имя клиента или компания. Для опроса: **Poll** (или краткое название опроса) |

### 4.3. Строка 3 — Тип источника

| Колонка | Содержимое |
|---------|------------|
| 1–5 | Пусто |
| 6 | Пусто |
| 7+ | `interview` / `paid` / `trial` / `Poll` |

Если клиент платил (pilot, POC, оплата) — укажи `paid` или `trial`. Иначе — `interview`.

### 4.4. Строка 4 — Заголовки фиксированных колонок

Соответствует шаблону: `;;Number;Budget;Need;;[client1 budget];[client2 budget];...`

| Кол 1 | Кол 2 | Кол 3 | Кол 4 | Кол 5 | Кол 6 | Кол 7+ |
|-------|-------|-------|-------|-------|-------|--------|
| (пусто) | (пусто) | Votes | Budget | Need | (пусто) | [budget клиента 1], [budget 2], … |

Бюджет по клиенту — если известен из call-database (pilot, POC, оплата). Иначе — пусто.

## Step 5. Извлечение потребностей и группировка

### 5.1. Потребности из call-database

Для каждого **NEED-XXX** извлеки:
- **Need text:** «Я хочу [need]» (из заголовка NEED-XXX)
- **Need Category:** из поля **Need Category** — для группировки в секции
- **Frequency (Частота):** общее число проголосовавших/подтвердивших
- **Budget:** если в call-database есть информация о бюджете (pilot, POC, оплата) — укажи; иначе — пусто
- **Taxonomy level:** T, M, D, N, U — по genotype-needs-taxonomy (можно вывести из Приоритет: HIGH→M/T, MEDIUM→D, LOW→N)

### 5.2. Группировка по секциям и порядок следования потребностей

Сгруппируй потребности по **Need Category**. Каждая группа = секция с заголовком `[NEEDS SECTION NAME]`.


#### Создание секций с нуля

Если в файле genotype.md пока нет секций, либо в мы обрабатываем много новых потребностей, которые еще не были зафиксированы в этом файле, то надо создать секции.
Проанализируй все потребности и придумай 3-5 секций как их сгруппировать.
Например можно выделить подгруппу потребностей юзкейсы (крупные потребности описывающие зачем и как клиент хочет использовать продукт), отдельные группы потребностей для рызных категорий пользователей, отдельные секции по модулям вашего продукта, нефункциональные требования и т.п. Исходи рационально из каждого конкретного продукта, выбери подход, который максимально уместен согласно числу и набору требований. Используй **небольшое число секций** (целевой ориентир — до пяти). Не создавай множество мелких секций. Все потребности, которые не попадают в явные категории, размещай в этих же пяти секциях — не создавай для них отдельные группы.

#### Добавление в в Genotype файл с существующей структурой секций

Если в файле genotype.md есть секции, стараться придерживаться их, добавить в них потребности которые еще не добавлены. Если есть очень много новых потребностей, которые еще не были добавлены в файл, то возможно будет целесообразно добавить 

#### Сортировка потребностей внутри секций

Потребности должны идти внтури секций от самой частой потребности (больше всего голосов) к менее частым.

Таким образом, нумерация потребностей NEED-001, NEED-002, NEED-003 никак не влияет на порядок, не нужно выводить их в порядке этой нумерации. Вместо этого нужно стараться помещать потребности в секции и внутри секции сортировать их по убыванию частоты (чилса голосов)

Строки потребностей должны быть сгруппированы по секцийм.

## Step 6. Заполнение строк потребностей

### 6.1. Формат строки

Для каждой потребности (соответствует шаблону `;I want [need];[number];[budget];[taxonomy];;[0 or 1];...`):

| Кол 1 | Кол 2 | Кол 3 | Кол 4 | Кол 5 | Кол 6 | Кол 7+ |
|-------|-------|-------|-------|-------|-------|--------|
| (пусто) | I want [need] | Число voters | Сумма budget | T/M/D/N/U | (пусто) | Vote count per source |

Для строки-секции (заголовок группы): `[NEEDS SECTION NAME];;;;;;;...` — колонка 1 содержит имя секции, остальные пусты.

### 6.2. Колонка 3 — Votes (всего клиентов заинтересованных в данной функции)

**Суммарное число** людей, которые:
- подтвердили потребность в интервью (1-on-1), или
- проголосовали в опросе (N человек)

Используй поле **Частота** из call-database. Если указано «1 из интервью + 7 по опросу» → 8.

### 6.3. Колонка 4 — Budget

Если известен бюджет (клиент платил, pilot, POC) — суммируй бюджеты всех, кто проголосовал за эту потребность. Иначе — оставь пустым.

### 6.4. Колонки 7+ — Vote count per source

Для каждого источника (интервью/опрос) в соответствующей колонке:

| Тип источника | Значение |
|---------------|----------|
| Интервью, потребность подтверждена | `1` |
| Интервью, потребность не подтверждена | (пусто) |
| Опрос, N человек проголосовало | `N` (число) |
| Опрос, «несколько человек» | `?` или оставь пусто |

Если точное число известно приблизительно (например 3-4 человека), то пишем цифру 3 

**Правило:** если в Source Calls для данной потребности есть запись с этим источником — заполни ячейку. Если нет — оставь пусто.

**Объединённая колонка опросов (одна дата):** если несколько опросов сведены в одну колонку (см. 3.2), суммируй голоса по всем этим опросам для данной потребности и укажи сумму в ячейке.

---

## Step 7. Формат CSV

### 7.1. Разделитель

Используй **точку с запятой** (`;`) как разделитель (как в шаблоне) — для корректного открытия в Excel с разными локалями.

### 7.2. Экранирование

Если в ячейке есть `;` или перевод строки — заключи в кавычки.

### 7.3. Кодировка

UTF-8 с BOM — для корректного отображения кириллицы в Excel.

---

## Step 8. Проверка перед сохранением

Перед записью файла убедись:

- [ ] **Целевой файл:** `student-projects/[student-name]/genotype/genotype.csv` (не .md)
- [ ] Путь содержит правильный `[student-name]`
- [ ] Колонки источников: слева — самые новые, справа — самые старые
- [ ] Опросы на одну дату без различающей информации — в одной колонке (не разбивать по вопросам)
- [ ] Строки 1–4 — заголовки по шаблону
- [ ] Все потребности из call-database перенесены, сгруппированы по Need Category
- [ ] Votes = сумма voters, Budget = сумма известных бюджетов
- [ ] Никакие строки не сбились, нет пропущенных ячеек
- [ ] Внутри секций потребности отсортированы по убыванию Votes (если нет, то пересортировать)
- [ ] В ячейках источников: 1 для интервью, N для опроса, пусто если не подтверждено
- [ ] Формат CSV: `;` разделитель, UTF-8

## Output

- **Созданный или обновлённый файл:** `student-projects/[student-name]/genotype/genotype.csv`
- **Сообщение пользователю:** Кратко опиши, что сделано: сколько потребностей, сколько источников (интервью/опросов), на основе call-database.

---

## Quality Criteria

**Полнота:**
- Все потребности из call-database отражены в genotype
- Все уникальные источники (интервью и опросы) представлены колонками

**Корректность:**
- Votes = сумма подтверждений/голосов
- Vote count в ячейках соответствует Source Calls
- Порядок колонок: от новых к старым

**Согласованность:**
- Структура соответствует genotype-template.csv
- Need Category используется для группировки секций
